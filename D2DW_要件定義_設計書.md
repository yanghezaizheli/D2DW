# D2DW 戸別訪問記録 PWA 要件定義・設計書

- バージョン: v0.1（1区域プロト向け）
- 作成日: 2026-06-23
- 対象: 戸別訪問（戸別伝道）の訪問記録を管理するPWA

---

## 1. 目的・前提

戸別訪問の記録を、スマホからカード操作で素早く残し、複数人でリアルタイムに共有することを目的とします。現行のGoogleスプレッドシート＋GAS＋地図PDF＋マスター台帳は「データの入口＋バックアップ」として残し、運用の正本（マスターデータ）はPWA専用DBへ移します。

決定済みの前提:

- 利用規模: 約70人。同時接続は70を超えない想定。総レコード約5万行。
- 同期方針: リアルタイム同期を優先。常時オンライン前提で、電波断時は自動再接続。
- 地図: 自分の現在地を地図に表示し、ブロックのピンから詳細地図に入る。可能なら詳細地図（PDF）上の各戸タップで記録入力。

---

## 2. 技術構成

| 層 | 採用 | 理由 |
|---|---|---|
| フロント | PWA（React + Service Worker 想定） | ホーム追加・キャッシュ・スマホ最適化。フレームワークは好みで調整可。 |
| バックエンド/DB | Supabase（PostgreSQL + Realtime + Auth + RLS） | 70同時接続が無料枠（200）内。SQLで集計が素直。行レベルの権限管理が可能。 |
| 地図 | Leaflet + OpenStreetMap | 無料・APIキー不要。現在地・ピン・画像オーバーレイ（詳細地図）まで1ライブラリで実現。 |
| ホスティング | 静的ホスティング（Vercel / Netlify / Cloudflare Pages 等） | PWA配信に十分。無料枠で開始可能。 |

補足: 「オフライン最優先」なら Firebase Firestore（オフライン永続化が標準）も有力でしたが、今回は「リアルタイム優先・常時オンライン」のため Supabase を採用します。

---

## 3. 非機能要件（無料枠での試算）

- 同時接続: 70 ＜ Supabase無料枠 200。✓
- データ量: 5万行 × 約1KB ≒ 最大50MB ＜ 無料DB 500MB。✓
- リアルタイム: 月200万メッセージ無料に対し、5万件規模の更新は余裕。✓
- → 当面は無料（$0）で運用できる見込み。メッセージや容量が急増した場合のみ Pro（$25/月、接続500・メッセージ500万）へ拡張。
- 電波断対応: Supabase Realtime は切断時に自動再接続。送信中の記録は端末側で一時保持し、再接続後に送信して取りこぼしを防ぐ。

## 4. データモデル

### blocks（区域）
| 列 | 内容 |
|---|---|
| id | 主キー |
| code | 区域番号 |
| name | 区域名（例: 01長住2） |
| area | エリア |
| lat / lng | 広域マップ用の代表座標 |
| wide_map_url | 広域地図PDFのURL |
| detail_map_url | 詳細地図PDFのURL |

### places（戸・建物・号室を統一管理）
| 列 | 内容 |
|---|---|
| id | 主キー |
| block_id | 所属区域 |
| kind | 種別（戸建て / 集合住宅 / 号室） |
| parent_id | 号室の場合は親（集合住宅）のid |
| label | 号 |
| display_name | 表記・建物名 |
| address | 住所 |
| map_x / map_y | 詳細地図上のタップ座標（任意・第2段階で付与） |
| status | 現在ステータス（例: 訪問可能） |
| note | 注記 |

### visits（訪問ログ：1タップ＝1行）
| 列 | 内容 |
|---|---|
| id | 主キー |
| place_id | 対象の戸・号室 |
| user_id | 記録者 |
| outcome | 会えた / 不在 / 投函 |
| visited_at | 記録日時（タイムスタンプ） |
| note | メモ |

### users（利用者） / assignments（区域担当・任意）
- users: id / name / email / role
- assignments: user_id / block_id（誰がどの区域を担当するか）

### カードに出す導出項目（visits から自動計算）
| 表示項目 | 算出方法 |
|---|---|
| 現在ステータス | places.status（手動）＋ is_due（自動判定）の組み合わせ |
| 前回訪問日時 | max(visited_at) |
| 前回面会日時 | max(visited_at where outcome = 会えた) |
| 累計不在回数 | count(outcome = 不在) |
| 前回結果 | 最新 visited_at の outcome |
| 次回訪問可日時 | last_visit_at ＋ タイプ別間隔（前回結果に応じて選択） |
| 訪問可能か(is_due) | status='訪問可能' かつ 次回訪問可日時を過ぎている |

> 設計意図: 表示値をすべて訪問ログから集計するため、要件の各指標が自動でそろい、履歴も残ります。現行スプシに無かった「前回訪問日時」もこの方式で自然に得られます。

### ステータス自動更新ルール（決定）

- 手動・固定ステータス: `訪問拒否`（赤）/ `他言語` / `JW` / `転居・空家`（灰）。人が手で設定し、時間では変化しない。`is_due` を常に false にして対象から外す。
- 既定の `訪問可能` は、前回訪問の結果とタイプ別の再訪間隔から「今訪問してよいか(is_due)」を自動算出する。間隔を過ぎたら自動で訪問可能（緑）に戻り、過ぎるまでは「対象外（次回訪問可 M/D）」表示。
- 再訪間隔は `visit_rules` 表で管理（暫定値・ST-M等は要確認）:

| タイプ | 不在後 | 投函後 | 会えた後 |
|---|---|---|---|
| LDR（戸建てエリア） | 30日 | 30日 | 90日 |
| ST-M / EV-M / AL-M（集合住宅系） | 90日 | 30日 | 180日 |

- 有効タイプの決定順: 建物個別 `places.type` ＞ 区域 `blocks.type` ＞ 種別からの既定（戸建て=LDR ／ 集合住宅・号室=EV-M）。
- 算出は DB の `place_stats` ビューで行い、アプリは `is_due` / `next_visitable_at` を読むだけ。区域カード・ピン・号室の集計は「今訪問可」件数を表示する。

## 5. 画面構成・遷移

| # | 画面 | 内容 | 主な操作 |
|---|---|---|---|
| 1 | 広域マップ | 現在地（青ドット）＋区域ブロックのピン | ピンタップ → 画面2 |
| 2 | 区域詳細 | 詳細地図(PDF)＋各戸カード一覧 | 各戸タップ → 画面3 ／ 集合住宅タップ → 画面4 |
| 3 | 記録入力 | 「会えた／不在／投函」を選び保存 | 保存 → 画面2へ戻る・即同期 |
| 4 | 号室一覧 | 集合住宅の号室をカードで並べる | 号室タップ → 画面3（記録方法は各戸と共通） |

カード表示（各戸・号室で共通）: 表記 ／ 現在ステータス（色分け）／ 前回訪問日時 ／ 前回面会日時 ／ 累計不在回数。

## 6. 地図設計

広域マップ（画面1）:

- Leaflet + OpenStreetMap タイル。`navigator.geolocation` で現在地を青ドット表示。
- 台帳の区域ごと緯度経度でブロックピンを配置。ピン色＝進捗（残り件数など）。

詳細マップ（画面2）:

- Leaflet（`CRS.Simple`）で詳細地図PDFを画像レイヤーとして表示。
- 各戸を画像座標 `(map_x, map_y)` のマーカーで配置し、タップで記録入力（理想形）。
- 集合住宅は建物マーカー → タップで号室一覧（画面4）。号室は座標が重なるため地図化せずカードリスト。

段階導入（「戸タップ」は座標登録が前提のため）:

- 第1段階: 詳細地図はPDF表示＋各戸カードリストで記録（座標登録なしで即運用開始）。
- 第2段階: 区域ごとに管理画面でPDF上をクリックして戸位置を登録 → 「地図上の戸タップ」を有効化。

> 理由: 各戸の `(map_x, map_y)` 登録には一度作業が必要なため、まず動かしてから区域単位で段階的に付与すると、止まらずに運用へ移れます。

## 7. 既存システムの活かし方

- 初期インポート: 台帳xlsx＋各区域スプシ → Supabase へ1回移行（移行スクリプトを用意）。
- バックアップ／共有閲覧: 定期エクスポート（スケジュール実行）でスプシへ書き戻し、従来どおりスプシでも閲覧可能に。
- 地図PDFはそのまま詳細地図として活用。
- 位置づけ: 正本はPWA専用DB、スプシは「入口＋バックアップ」。

---

## 8. 認証・権限

- Supabase Auth で **Googleログインのみ**。メール許可リスト（`allowed_emails`）に載るアカウントだけが利用可（`is_allowed()` をRLSとアプリ双方でチェック）。
- RLS（行レベルセキュリティ）で区域担当のアクセス制御（assignments を利用）。70人規模での誤操作・他区域の誤編集を防止。

---

## 9. 段階リリース計画

1. プロト: 1区域（例: 01長住2）をダミーデータで画面1〜4を通し動作確認。
2. データ移行: 該当区域を Supabase へインポートし、実データで検証。
3. 横展開: 全区域をインポート、認証・権限を有効化。
4. 地図高度化: 詳細地図の戸ホットスポット登録、現在地連動を最適化。
5. 運用: スプシ自動エクスポート、必要に応じ Pro へ移行。

## 11. 確認・未決事項

- ログイン方式 → 決定済み（§8）: Google認証のみ＋メール許可リスト（allowlist）。
- 種別コードの正式な意味（推測: LDR=戸建てエリア、ST-M=階段式、EV-M=エレベーター式、AL-M=オートロック式の集合住宅）。※ visit_rules の ST-M 値が要確認。
- ステータス自動更新ルール → §4で決定済み（手動固定ステータス＋タイプ別間隔による is_due 自動判定）。
- 詳細地図の戸ホットスポット登録を誰がどう行うか（運用フロー）。
- フロント技術 → 決定済み: 素のHTML/JS＋Leaflet（単一HTMLファイル, Supabase JSクライアント）。
